Treez — 82% Less Design Debt, 30% Faster QA

Building Root Design System to unify a fragmented B2B platform, cutting onboarding from 3 weeks to 1 and enabling cross-platform scale.

Overview

82% less duplication. 30% fewer QA tickets. Designer onboarding cut from 3 weeks to 1.

Treez is the mission-critical OS for enterprise cannabis retail. When I joined as Design System Lead, rapid growth had created a “re-invention tax”—teams rebuilt the same UI every sprint with different quality. The lack of unified design language was blocking mobile expansion.

Over 12 months, I architected Root Design System, transforming fragmented assets into a scalable foundation that enabled cross-platform development and enterprise accessibility compliance.

“It’s the first time we’ve had a true source of truth.” — Sam, Engineering Lead

The Problem

Scaling Chaos

The product looked coherent on the surface, but underneath was a “Wild West” preventing scale. As we prepared to launch mobile POS in 2024, the fragmented codebase made every feature exponentially expensive.

Visual Fragmentation

Inconsistent patterns eroded user trust. 20+ undocumented grey styles drifted across surfaces.

Zero Source of Truth

No single reference existed. 100+ detached patterns caused ~60% of effort rebuilding existing UI.

Compliance Risk

Critical exposure: 80% of core interactions failed WCAG 2.1 AA, threatening enterprise contracts.

“Without a system, we were rebuilding the same UI — with different quality — every sprint.”

The Approach

From Audit to Architecture

We moved from reactive design to a strategic roadmap:

High-level Process

Phase 1: Discovery

  • Visual Debt Audit
  • Compliance Assessment
  • Stakeholder Alignment

Phase 2: Foundation

  • Token Architecture
  • Core Components
  • Documentation Portal

Phase 3: Scale

  • Pattern Library
  • Contribution Model
  • Mobile Parity
Continuous adoption tracking throughout all phases

Why this sequence? Mobile POS was blocked by accessibility gaps. By front-loading compliance into the token foundation, we unlocked both immediate cleanup AND the strategic platform expansion.

Key Decisions

Trade-off: Depth vs. Breadth

We chose to build deep governance infrastructure rather than a broad component library. Root wasn’t just components—it was a shared ownership model.

Rationale: Treez teams had tried component libraries before. They failed because Design and Engineering spoke different languages. We needed to fix the handoff, not just the assets.

The Bridge (Portal)

Co-developed internal portal mapping Figma tokens directly to CSS variables—1:1 implementation parity.

Bi-directional Contribution

Established governance allowing both designers and developers to evolve the system responsibly.

Documentation as Product

Connected Figma → Zeroheight → Storybook. Single source of truth across all disciplines.

Source of Truth Portal: Colors and Buttons

Results

The Numbers

Root created clarity and scale across the Treez ecosystem.

82% Less Figma Duplication

Centralized assets eliminated redundant files, streamlining design workflows.

30% Fewer QA Tickets

Consistent components reduced visual discrepancies and implementation errors.

Onboarding: 3 Weeks → 1 Week

Comprehensive documentation accelerated new team member ramp-up.

94% A11y Issues Resolved

Built-in compliance ensured the product met enterprise standards.

Reflection

Shared Ownership Over Individual Craft

Root wasn’t just a project—it was a shift in how Treez teams build, collaborate, and scale. We moved from individual craftsmanship to shared ownership. From design chaos to clarity.

What I’d do differently: Start accessibility linting in CI/CD earlier. By month 8, the system was catching issues that could have been automated from day one.

Key learning: Design Systems fail when they’re “designed for” engineering. Root succeeded because it was “designed with” engineering—the portal and contribution model were co-created from sprint one.
